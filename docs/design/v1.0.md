# 🚢 Shipyard v0.1 Design

> **Shipyard is a project lifecycle companion that manages the information surrounding your source code.**

Developers write software.

Shipyard manages everything around the software.

Every software project eventually accumulates project metadata—roadmaps, changelogs, documentation, release notes, TODOs, and other files that are essential to maintaining the project but are not part of the source code itself. Keeping these files synchronized quickly becomes repetitive and easy to forget.

Shipyard exists to solve that problem.

Rather than replacing existing tools such as Git, Poetry, MkDocs, or GitHub Actions, Shipyard works alongside them. It understands how they fit together, manages project metadata, automates repetitive maintenance, and coordinates the project lifecycle while leaving each tool responsible for its own job.

Developers build software.

Shipyard manages everything around the software.

---

# Philosophy

Shipyard is built around a few simple principles.

* Markdown is the source of truth.
* Project metadata should live inside the repository.
* Existing tools should be integrated, not replaced.
* Developers should remain in complete control.
* Every piece of project information should have a single place to live.

Shipyard is not a package manager.

Shipyard is not a build system.

Shipyard is not a CI/CD platform.

Shipyard is a project lifecycle engine that manages the information surrounding a software project.

---

# Design Decisions

Whenever possible, Shipyard follows established community standards instead of inventing new ones.

## Changelog Standard

Shipyard follows the **Keep a Changelog** specification.

Reference:

* https://keepachangelog.com/en/1.1.0/

Every roadmap feature belongs to one of the standard changelog categories:

* Added
* Changed
* Deprecated
* Removed
* Fixed
* Security

These categories are preserved from planning all the way to release generation.

---

## Versioning Standard

Shipyard follows **Semantic Versioning (SemVer 2.0.0)**.

Reference:

* https://semver.org/

Version numbers follow the format:

```
MAJOR.MINOR.PATCH
```

Examples:

```
0.1.0
0.2.0
1.0.0
1.4.2
```

General rules:

* Patch releases contain backwards-compatible bug fixes.
* Minor releases contain backwards-compatible features.
* Major releases contain breaking changes.

Shipyard uses Semantic Versioning when organizing roadmap versions and generating releases.

---

## Why Shipyard does not use Git commit messages

Commit history and release history serve different purposes.

Git commits describe how development happened.

Changelogs describe what changed for users.

Commit messages frequently contain:

* work in progress
* temporary changes
* refactoring
* debugging
* incomplete implementations

For this reason Shipyard generates changelogs from completed roadmap items and additional release changes instead of raw Git history.

# Shipyard v0.1 Scope

Version 0.1 focuses on one responsibility:

> **Managing project metadata throughout the development lifecycle.**

The first release intentionally keeps the scope small and concentrates on the information that surrounds the source code.

---

# 1. Roadmap

The roadmap is the project's release plan.

It stores planned features grouped by release versions.

Each roadmap entry represents a **feature**, not implementation work.

Example:

```md
## 🚀 v0.1.0

- [ ] SHP-001 added: Generate roadmap
- [ ] SHP-002 added: Generate changelog
- [ ] SHP-003 changed: Replace Rich with curses
- [ ] SHP-004 fixed: Markdown parser bug
- [ ] SHP-005 security: Validate file paths
```

Because every roadmap item already belongs to a changelog category, Shipyard can later generate release notes automatically.

---

# 2. Additional Changes

Not every change is planned.

During development developers frequently discover bugs, small improvements, or cleanup work that was never part of the roadmap.

These changes are stored separately.

Example:

```md
## Additional Changes

- fixed: Terminal flickering
- fixed: Memory leak in parser
- changed: Faster CLI startup
```

During a release these entries are merged into the generated changelog before the section is cleared.

---

# 3. Working Tasks

A roadmap feature is usually too large to complete in a single coding session.

Instead of polluting the roadmap with implementation details, each roadmap feature can have its own working task list.

Example:

```
SHP-003

Replace Rich with curses

- [x] Create adapter interface
- [x] Rich backend
- [ ] Curses backend
- [ ] Handle terminals without color
- [ ] Remove Rich dependency
```

Working tasks describe **how** a feature is implemented.

They never appear in the generated changelog.

When all working tasks are completed, Shipyard can offer to mark the corresponding roadmap feature as complete.

---

# 4. Current Feature

Shipyard remembers which roadmap feature is currently being worked on.

Example:

```
Current Feature

SHP-003
Replace Rich with curses
```

Running:

```
shipyard current
```

shows the active feature together with its remaining implementation tasks, making it easy to resume work after closing the project or switching machines.

---

# 5. Changelog Generation

Shipyard generates changelogs from two sources:

* completed roadmap features
* additional changes

Generated changelogs follow the Keep a Changelog structure:

```
Added
Changed
Deprecated
Removed
Fixed
Security
```

Released changelog sections are intended to be generated automatically rather than maintained manually.

---

# 6. Metadata Registry

Shipyard maintains a registry of project metadata files.

Examples include:

* README.md
* ROADMAP.md
* CHANGELOG.md
* documentation entry files

These files may contain Shipyard placeholders.

During synchronization Shipyard updates only the registered sections, leaving the remainder of each document untouched.

---

# 7. Ideas

Ideas are intentionally separate from the roadmap.

They represent proposals, experiments, and future possibilities that are not yet scheduled for implementation.

Ideas are stored as standalone documents.

Example:

```
docs/proposals/

    obsidian.md
    workflows.md
    github.md
```

Roadmap entries may reference proposal documents.

Example:

```md
- [ ] SHP-014 added: Obsidian Adapter (proposal: docs/proposals/obsidian.md)
```

This keeps the roadmap concise while allowing complex ideas to evolve independently.

---

# Information Lifecycle

Every piece of project information moves through a single lifecycle.

```
Ideas

↓

Roadmap

↓

Current Feature

↓

Working Tasks

↓

Development

↓

Additional Changes

↓

Release

↓

Changelog

↓

Project Metadata
```

Each stage has a single responsibility.

Ideas become roadmap features.

Roadmap features become implementation work.

Implementation produces releases.

Releases update project metadata.

Nothing is duplicated, and every piece of information has one canonical location.

---

# Responsibilities

## The Developer

* Writes source code.
* Designs the application.
* Makes technical decisions.

## Shipyard

* Organizes project metadata.
* Maintains the roadmap.
* Tracks implementation progress.
* Generates changelogs.
* Synchronizes project metadata.
* Coordinates project maintenance.
* Integrates with existing development tools.

Shipyard does **not** replace Git, Poetry, MkDocs, GitHub Actions, or other development tools.

Instead, it acts as the project companion that connects them together while keeping the project's information organized throughout its lifecycle.
